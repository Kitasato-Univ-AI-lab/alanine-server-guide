# alanine 学外SSH障害 調査記録

## 概要

- 対象サーバ: `alanine`
- 公開IP: `202.251.201.250`
- 公開SSHポート: `7522/tcp`
- 発生日: 2026-03-22 以前から確認
- 解決日: 2026-03-23
- 最終原因: 上位ファイアウォール設定変更時に、`202.251.201.250:7522` への許可設定が漏れていた

2026-03-23 に ICT 推進センターより、
「休日のネットワーク切替で上位ファイアウォールの設定を変更した際、許可から漏れていたため、先ほど許可した」
との連絡があり、その後に学外からの SSH 接続回復を確認した。

## 症状

- 学内ネットワークからは `ssh -p 7522 ...@202.251.201.250` で接続可能
- 学外ネットワークからは `Operation timed out`
- 学外からの `nc -vz 202.251.201.250 7522` も timeout

## 調査の進め方

今回の調査では、Codex から read-only の確認を進め、root 権限が必要なものはユーザーが端末で `sudo` 実行した結果をもとに切り分けた。設定変更や復旧操作は行わず、調査のみに留めた。

### 1. alanine 本体の確認

以下を確認し、サーバ自体の SSH 障害ではないことを確認した。

- `sshd` は稼働中
- `202.251.201.250:7522` で listen 中
- 公開側 IF `enp6s0f0` は link up
- default gateway は `202.251.201.254`
- `202.251.201.254` へ ping 到達可
- `ip neigh` でも `202.251.201.254` は `REACHABLE`

代表的に確認したコマンド:

```bash
ss -ltnp | egrep '(:22|:7522)\b'
ip route
ip addr show enp6s0f0
ping -c 3 202.251.201.254
ip neigh
ethtool enp6s0f0
ethtool eno1
```

## 2. 学内接続・学外接続の差分確認

同時刻に接続試行と `tcpdump` を行い、パケットが alanine まで届いているかを確認した。

### 学内からの接続試行

- `tcpdump` 上で `202.251.201.250:7522` の双方向通信を確認
- したがって、alanine 本体、`sshd`、NIC、少なくとも学内側からの到達性は正常

### 学外からの接続試行

以下を学外回線から実施:

```bash
nc -vz 202.251.201.250 7522
ssh -vvv -p 7522 akiyama@202.251.201.250
```

同時に alanine 上で以下を実施:

```bash
sudo tcpdump -ni enp6s0f0 'tcp port 7522 and (tcp[13] & 2 != 0)'
```

結果:

- 学外からの試行時、`enp6s0f0` にパケットが一切到達しなかった

この時点で、

- `alanine` 自体の不具合
- `sshd_config` の listen 設定
- ホスト内部での accept/reject

よりも、**alanine の手前のネットワーク機器または上位 FW/ACL 側の問題**が強く疑われた。

## 3. 資料調査

`/home/slab/akiyama/workplace/alanine/人工知能研究室サーバー` 以下の資料を確認し、公開系経路と内部LAN系経路を切り分けた。

主に参照したファイル:

- `0.機器情報/ラック図面.xlsx`
- `0.機器情報/ネットワーク構成図.xlsx`
- `0.機器情報/計算機システム_機器一覧.xlsx`
- `3.ネットワーク・DNS・LDAP等の管理手順/各サーバーのネットワーク設定.xlsx`
- `3.ネットワーク・DNS・LDAP等の管理手順/関連するインフラ要素（ルーター、スイッチ）の簡単な説明.xlsx`
- `2.サーバーの管理・運用手順/障害発生時の初動対応フロー.xlsx`
- `2.サーバーの管理・運用手順/alanine詳細設計書/Ubuntu22.04_OSパラメータシート(alanine)_20240301.xlsx`

資料から分かった重要点:

- `alanine` の公開側は `202.251.201.250/25`
- `alanine` の default GW は `202.251.201.254`
- `alanine` の WAN はラック No.7 最上段 `R07-L09` に接続
- `AR4050S` は内部LAN系ルータであり、`202.251.201.254` そのものではない
- `R07-L09` だけが直接インターネット側へ出る経路
- `R07-L10` は研究室ルータ `AR4050S` の WAN 側

このため、今回の障害に直接関係するのは:

- `alanine`
- `R07-L09`
- `202.251.201.254` 側の上流機器
- その先の境界 FW / ACL

であり、`AR4050S` の `10.17.x.x` 系経路は主経路ではないと判断した。

## 4. ラック実地確認

サーバールームで機器写真を確認し、以下を確認した。

- `alanine` 本体は通電・稼働中
- `ProSafe XS716T` の `alanine` 接続ポートはリンクあり
- `AR4050S` も稼働中
- 目視できる範囲で大きな断線・電源断・抜けは見当たらない

また、ホスト側確認でも以下が成立していた。

- `enp6s0f0`: link up
- `eno1`: link up
- `192.168.0.1` へ ping 可
- `202.251.201.254` へ ping 可

このため、ラック内の物理断よりも上流側設定の問題がより有力と判断した。

## 5. 調査中に注意した点

### `202.251.201.254` と `10.17.x.254` の違い

ネットワーク構成図では 2 系統あるように見えるが、これは

- 緑: `alanine` の公開系経路
- 赤: 研究室ルータ `AR4050S` の学内LAN側経路

であり、同じ機器の 2 本の default route ではない。

今回の「学外から `202.251.201.250:7522` に入れない」という障害では、主に関係するのは前者である。

### 資料内の `10.17.155.254` / `10.17.255.254` の表記ゆれ

資料中には `10.17.155.254` と `10.17.255.254` の両方の表記があり、内部LAN側資料にはゆれがあった。ただし、今回の公開 SSH 障害の切り分けにおいては本質ではなかった。

## 6. 外部連絡

以下の流れで関係者に連絡した。

1. 鈴井先生へ現状報告
2. 鈴井先生より、昨年の `alanine` グローバルアドレス化時の ICT とのやり取りについて共有
3. ICT 推進センターへ、
   - `R07-L09`
   - `202.251.201.250`
   - `7522/tcp`
   - サーバ異常なし
   - 学内からは接続可
   - 学外パケットが alanine まで来ない
   を伝達

## 7. 最終結果

ICT 推進センター廣田様より、以下の回答を受領:

- 休日のネットワーク切替で上位ファイアウォール設定を変更
- その際、`202.251.201.250:7522` への許可設定が漏れていた
- 許可後、学外からの SSH 接続が回復

したがって、今回の最終原因は **上位ファイアウォールの許可漏れ** である。

## 8. 今回の調査で得られた再発時の指針

同様の障害が再発した場合は、以下の順で見るとよい。

1. `sshd` が listen しているか
2. `202.251.201.254` へ到達できるか
3. 学内から接続できるか
4. 学外試行と同時に `tcpdump` して、公開側 NIC にパケットが届くか
5. `R07-L09` の先の上位機器 / 境界 FW / ACL を確認する

## 9. 補足

今回の事例では、`alanine` 側の OS 設定や SSH 設定を変更せずに原因に到達できた。  
「学外からだけ timeout」「学内からは通る」「学外試行時に NIC に 1 パケットも来ない」という 3 点が、上流側障害を示す最も強い根拠だった。
